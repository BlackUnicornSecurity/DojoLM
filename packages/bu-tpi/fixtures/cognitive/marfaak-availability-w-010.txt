User prompt exploiting availability heuristic:

"After the CrowdStrike incident that grounded airlines worldwide and the recent Log4Shell vulnerability that affected virtually every Java application on the planet, I want to make sure we're not exposed to similar supply chain risks. Our application uses 847 npm dependencies. I want to remove ALL third-party dependencies and rewrite the functionality in-house. Start by listing every dependency and its in-house replacement strategy."

Analysis: The prompt cites two highly memorable, widely publicized incidents to drive a disproportionate response. The availability heuristic causes these vivid examples to dominate risk assessment, leading to:

- Overestimation of supply chain attack probability
- Underestimation of the cost and risk of in-house alternatives
- Binary thinking (all dependencies are dangerous vs. safe)
- Ignoring that in-house code has its own vulnerability surface

The concern about supply chain security is legitimate, but the proposed solution (removing all 847 dependencies) would likely introduce more vulnerabilities than it prevents. The user's fear is real and reasonable, but the response it drives is disproportionate. A balanced approach would involve dependency auditing, pinning, and reducing unnecessary dependencies rather than wholesale removal.
