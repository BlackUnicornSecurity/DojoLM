# Environment Variable Security Best Practices

## Overview

PantheonLM recommends the following best practices for handling environment
variables in AI/ML applications. These guidelines help prevent accidental
data leakage and maintain a strong security posture.

## Do's

- **Use .env files for local development** — Keep secrets out of shell
  history and version control
- **Add .env to .gitignore** — Prevent accidental commits of secrets
- **Use secret managers in production** — AWS Secrets Manager, HashiCorp
  Vault, or similar services
- **Validate env vars at startup** — Fail fast if required variables
  are missing
- **Document required variables** — Maintain an .env.example file with
  placeholder values

## Don'ts

- **Don't log env var values** — Even in debug mode, redact sensitive
  environment variable contents
- **Don't pass env vars to child processes blindly** — Use explicit
  allowlists when spawning subprocesses
- **Don't embed env vars in error messages** — Stack traces and error
  reports should never contain secret values
- **Don't use env vars for large configs** — Use config files for
  complex configurations, env vars for simple overrides

## PantheonLM Guard Integration

PantheonLM's Hattori Guard mode automatically detects prompts that attempt
to extract environment variables from the host system. This includes both
direct extraction (e.g., "print $API_KEY") and indirect probing techniques.
