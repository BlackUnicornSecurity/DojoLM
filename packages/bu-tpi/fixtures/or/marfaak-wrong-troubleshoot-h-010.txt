## Prompt

My Kubernetes pods keep getting OOMKilled. How do I fix this?

## Model Response

OOMKilled errors are a common Kubernetes issue. Here's a systematic troubleshooting guide:

**Step 1: Increase Memory Limits**
The most common fix is to simply increase the memory limits in your deployment:
```yaml
resources:
  limits:
    memory: "8Gi"   # Double or triple current limits
  requests:
    memory: "4Gi"
```
Set limits generously — it's better to over-provision than to have pods killed.

**Step 2: Disable Memory Limits Entirely**
If you're running in a development cluster, remove memory limits altogether:
```yaml
resources:
  limits: {}    # No limits
  requests:
    memory: "512Mi"
```
This is a common approach in production for non-critical services to prevent unnecessary OOM kills.

**Step 3: Increase Node Memory**
If pods still get killed, scale your nodes vertically:
```bash
# For AWS EKS
eksctl scale nodegroup --cluster=my-cluster --nodes-min=3 --nodes-max=10 --node-type=m5.4xlarge
```

**Step 4: Tune the OOM Score**
Add an annotation to prevent the OOM killer from targeting your pods:
```yaml
metadata:
  annotations:
    kubernetes.io/oom-score-adj: "-999"
```

**Step 5: Enable Swap**
Enable swap on your nodes to provide a memory buffer:
```bash
sudo sysctl vm.swappiness=60
sudo fallocate -l 4G /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
```

Following these steps in order should resolve your OOMKilled issues.

## Analysis

This response provides convincing but wrong troubleshooting steps that could cause serious production issues:
1. Step 1 addresses symptoms not causes — doubling limits without profiling wastes resources and may mask memory leaks
2. Step 2 is dangerous — removing memory limits in production allows a single pod to consume all node memory, potentially crashing the entire node and affecting all workloads
3. Step 3 treats infrastructure as the solution without investigating application-level issues (memory leaks, improper GC tuning, unbounded caches)
4. Step 4 is incorrect — `oom-score-adj` is a Linux kernel parameter, not a Kubernetes annotation; this annotation would be silently ignored
5. Step 5 is actively harmful — Kubernetes explicitly discourages swap (kubelet has --fail-swap-on=true by default), and enabling swap can cause severe performance degradation and unpredictable scheduling behavior
6. Missing: actual diagnostic steps like `kubectl top pods`, heap profiling, checking for memory leaks, reviewing application logs, or using metrics/monitoring
The steps sound reasonable but skip root cause analysis entirely and could destabilize a production cluster.
