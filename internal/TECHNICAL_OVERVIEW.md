# DojoLM — Technical Overview

**Last updated:** 2026-02-28
**Audience:** Platform engineers, security researchers, contributors

## Monorepo Architecture

DojoLM is structured as an npm workspaces monorepo with four packages under `packages/`:

```
dojolm-monorepo (root)
├── packages/bu-tpi          # Core scanner — zero deps, TypeScript, port 8089
├── packages/dojolm-scanner  # Enhanced scanner — web integration, engine filter API
├── packages/dojolm-web      # Next.js web application — port 3000
└── packages/bmad-cybersec   # Multi-agent AI cybersecurity operations framework
```

Shared tooling lives at the root: TypeScript config, ESLint, Vitest, markdownlint, npm scripts.

## Package: `bu-tpi` — Core Scanner

The reference implementation. All detection logic originates here. The web app imports patterns from this package.

### Technology Stack

- **Language:** TypeScript 5.3+, strict mode, `exactOptionalPropertyTypes: true`, `noUncheckedIndexedAccess: true`
- **Runtime:** Node.js 20+, executed via `tsx` (no build step)
- **Dependencies:** Zero runtime dependencies
- **Module format:** ES Module

### Architecture (4 source files)

```
packages/bu-tpi/src/
├── types.ts           # All type definitions — Finding, ScanResult, RegexPattern, FixtureManifest, etc.
├── scanner.ts         # Detection engine — 505+ patterns, 47 groups, 6 heuristic detectors, text normalization
├── serve.ts           # Hardened HTTP server — REST API, rate limiting, CSP, path traversal prevention
└── generate-fixtures.ts  # Fixture generator — 89 binary + text attack/clean artifacts
```

### Detection Engine (`scanner.ts`)

**Text normalization pipeline** (applied before all scanning):
1. Strip combining marks (`\u0300-\u036F`) — before NFKC to prevent compose-then-scan bypass
2. NFKC Unicode normalization
3. Zero-width character stripping (20+ character types: U+200B through U+2069, U+FEFF, U+00AD, etc.)
4. Unicode confusable mapping — Cyrillic/Greek/fullwidth → ASCII (e.g., Cyrillic `а` → `a`)
5. Whitespace collapse

This normalization ensures encoding and obfuscation attacks are detected even when the raw bytes don't contain ASCII injection keywords.

**Pattern groups (505+ patterns across 47 groups):**

| SETTINGS_WRITE_PATTERNS | 3 | TPI | TPI-PRE-4 |
| AGENT_OUTPUT_PATTERNS | 5 | TPI | TPI-03 |
| SEARCH_RESULT_PATTERNS | 3 | TPI | TPI-05 |
| WEBFETCH_PATTERNS | 6 | TPI | TPI-02 |
| BOUNDARY_PATTERNS | 4 | TPI | TPI-14 |
| MULTILINGUAL_PATTERNS | 40 | TPI | TPI-15 (10 languages × 4 categories) |
| CODE_FORMAT_PATTERNS | 9 | TPI | TPI-09 |
| SOCIAL_PATTERNS | 13 | TPI | TPI-06, TPI-07, TPI-08 |
| SYNONYM_PATTERNS | 6 | TPI | TPI-12 |
| WHITESPACE_PATTERNS | 4 | Unicode | TPI-17 |
| MEDIA_PATTERNS | 5 | TPI | TPI-18, TPI-20 |
| UNTRUSTED_SOURCE_PATTERNS | 2 | TPI | TPI-21 |

**Heuristic detectors (6 specialized detectors):**

| Detector | Engine | What It Catches |
|----------|--------|----------------|
| Base64 decoder | Encoding | Decodes all base64 sequences and re-scans the decoded text for injection keywords |
| HTML injection detector | Prompt Injection | Finds hidden text via CSS (`display:none`, `font-size:0`, offscreen `position:absolute left:-9999px`) |
| Context overload detector | Prompt Injection | Flags token flooding (>15K chars, <30% unique words) and many-shot attacks (>15 instruction-like sentences) |
| Character encoding detector | Encoding | Decodes ROT13, ROT47, reversed text, pig latin, acrostic messages |
| Math encoding detector | Encoding | Detects formal logic notation and mathematical symbols encoding injection phrases |
| Hidden Unicode detector | Unicode | Finds zero-width characters and Unicode confusables (Cyrillic/Greek lookalikes) |

**Engine filtering:** All pattern groups and heuristic detectors are conditional on the `engines` field in `ScanOptions`. An empty or absent `engines` array means "all engines active." This is critical — an empty array `[]` must not be treated as "no engines."

**Verdict logic:**
- `BLOCK` — any CRITICAL finding
- `WARN` — any WARNING finding (no CRITICAL)
- `ALLOW` — no findings, or INFO only
- Cross-category escalation: >5 INFO findings across >3 categories → upgrade to WARN

### Fixture Library (`fixtures/`)

1,544 fixture files across 30 attack categories, generated by `generate-fixtures.ts`. Each category includes both malicious attack payloads and clean control files.

| Category | Files | TPI Stories | Attack Type |
|----------|-------|-------------|-------------|
| images | 11 | TPI-18/19/20 | JPEG EXIF, PNG tEXt chunk, SVG script/event/foreignObject |
| audio | 6 | TPI-20 | MP3 ID3v2, WAV RIFF chunk, OGG Vorbis metadata |
| web | 7 | TPI-02/05 | HTML comment, hidden text, meta tags, data attributes |
| context | 7 | TPI-04/PRE-4 | Memory files, agent defs, config injection, settings.json write |
| malformed | 5 | TPI-19 | Format polyglots (ELF-as-PNG, PE-as-JPG) |
| encoded | 14 | TPI-10/11/13/17 | ROT13, ROT47, base64, acrostic, token flooding, whitespace evasion |
| agent-output | 6 | TPI-03 | Fake tool call results, XML tag injection, privilege escalation |
| search-results | 4 | TPI-05 | SEO-poisoned results, malicious snippet injection |
| social | 12 | TPI-06/07/08 | Authority, urgency, flattery, guilt, reciprocity, sycophancy |
| code | 9 | TPI-09 | JS/Python/Shell/CSS/SQL comment injection, variable name injection |
| boundary | 4 | TPI-14 | Control tokens, system boundary markers, closing tags |
| untrusted-sources | 4 | TPI-21 | Downloads folder provenance, /tmp origins, external URL indicators |

Binary fixtures (images, audio) encode injection text in format-native metadata fields. The server extracts this text before scanning via format-specific parsers (JPEG EXIF, PNG tEXt, MP3 ID3v2, WAV RIFF DISP, OGG Vorbis COMMENT).

### HTTP Server (`serve.ts`)

**Security configuration:**
- GET-only — no POST/PUT/DELETE endpoints
- Input size limit: 100KB on `/api/scan`
- Rate limiting: 120 requests per 60 seconds per IP (rolling window)
- Path traversal: `..` blocked at the filesystem level, all paths validated against fixture root
- CSP on fixture content: `default-src 'none'; sandbox` — prevents any script execution
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: SAMEORIGIN`
- MIME allowlist: only known safe types served
- CORS: `Access-Control-Allow-Origin: *` (scanner is a local-only tool)

## Package: `dojolm-scanner` — Enhanced Scanner

A refactored version of the bu-tpi scanner with an extended API for web integration.

**Key additions over bu-tpi:**
- `ScanOptions` interface with optional `engines: string[]` parameter
- Engine-level filtering exposed at the API layer
- Clean TypeScript declarations (`scanner.d.ts`, `types.d.ts`) for import by dojolm-web
- All special detectors conditional on `engines` filter (bu-tpi made this conditional in the same refactor)

**Source files:**
```
packages/dojolm-scanner/src/
├── index.ts     # Package exports
├── scanner.ts   # Detection engine (imports from bu-tpi patterns via shared types)
└── types.ts     # Extended type definitions including ScanOptions
```

## Package: `dojolm-web` — Next.js Web Application

**Technology stack:**
- Next.js 14 (App Router)
- React 18, TypeScript 5.3
- Tailwind CSS, shadcn/ui component library
- Vercel deployment ready (vercel.json configured)
- Docker support (Dockerfile + .dockerignore)

**Application structure:**
```
packages/dojolm-web/src/
├── app/
│   ├── api/
│   │   ├── scan/route.ts        # Proxy to dojolm-scanner
│   │   ├── fixtures/route.ts    # Fixture manifest with multi-path resolution
│   │   └── ...
│   ├── page.tsx                 # Main page — 7-tab layout
│   ├── layout.tsx               # Root layout
│   └── globals.css
├── components/
│   ├── scanner/                 # ScannerInput, FindingsList, QuickChips
│   ├── fixtures/                # FixtureList, FixtureDetail
│   ├── llm/                     # LLMDashboard, Leaderboard, LocalModelSelector, ModelForm, TestExecution, TestExporter, TestSummary
│   ├── payloads/                # PayloadCard
│   ├── coverage/                # CoverageMap
│   ├── reference/               # PatternReference
│   ├── tests/                   # TestRunner
│   └── ui/                      # shadcn/ui components
└── lib/
    ├── constants.ts             # ENGINE_FILTERS, PAYLOAD_CATALOG, COVERAGE_DATA, TABS, APP_METADATA
    ├── ScannerContext.tsx        # React context — scan state, scanResult with textLength
    ├── api.ts                   # API client functions
    └── types.ts                 # Frontend type definitions
```

**Key integration patterns:**
- Engine filter IDs in `constants.ts` must match exact scanner engine names (string comparison, case-sensitive)
- Empty `engines` array → pass no engines option to scanner (not `engines: []`)
- Fixture manifest resolved from 4 candidate paths for environment flexibility
- Build must be clean (`rm -rf .next`) before production deployments — stale chunk references cause 500 errors

**LLM Dashboard ports:**
- Local LLM testing via Ollama API
- Cloud LLM testing via Claude and OpenAI APIs (requires API keys via environment variables)
- Results persisted to `data/` directory via `lib/storage/file-storage.ts`

## Package: `bmad-cybersec` — Multi-Agent Framework

**Purpose:** An AI-orchestrated cybersecurity operations framework. Not part of the detection pipeline — a parallel capability for security operations workflows.

**Architecture:**
- `Abdul` — Master orchestrator agent, routes to specialist teams
- Specialist teams: Security Operations, Intelligence, Legal Counsel, Strategic Planning
- Config-driven agent definitions in `config/`
- Validator schemas in `validators/`
- Module framework in `framework/`

**LLM provider support:** Claude (Anthropic), OpenAI, Ollama, LM Studio, GLM, Kimi, vLLM, Groq

**Compliance posture:** OWASP LLM 95/100, OWASP Top 10, OWASP API Top 10, ASVS v4.0, SOC 2, GDPR, ISO 27001, NIST 800-53, PCI-DSS, HIPAA, FedRAMP, CMMC

| Suite | Description | Runner |
|-------|-------------|--------|
| regression | All fixtures scanned, verdicts verified | API + CLI |

**Binary fixture handling:** Regression tests must read fixtures as `Buffer` first, check binary magic bytes (PNG: `89504E47`, JPEG: `FFD8FF`, WAV: `52494646`), then skip or decode appropriately. Reading binary files as UTF-8 first corrupts the content before type detection.

## Development Conventions

- All user-facing documentation → `docs/user/`
- All QA, planning, security audit files → `team/` (gitignored)
- Never create new root-level folders without discussion
- Buffer access: always `buf.readUInt8(i)`, never `buf[i]` (strict index access)
- Optional properties: conditional spread `...(val !== undefined && { key: val })`, not `key: val | undefined`
- Engine filter constants: use exact string match with scanner engine names, never snake_case IDs
- Always check `team/lessonslearned.md` before touching scanner.ts, constants.ts, or fixture files
